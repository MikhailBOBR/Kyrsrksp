# Анализ предметной области

## Описание задачи

Тема проекта — персональный дневник питания с анализом КБЖУ. Пользователь ведет журнал приемов пищи, контролирует суточные цели по калориям, белкам, жирам и углеводам, отслеживает воду, самочувствие, динамику тела и планирует рацион на несколько дней вперед.

Практическая проблема предметной области состоит в том, что при ручном контроле рациона пользователь вынужден:

- собирать данные о продуктах из разных источников;
- отдельно считать КБЖУ по каждому приему пищи;
- сравнивать фактическое питание с дневной целью;
- держать в памяти режим питания, воду, самочувствие и сопутствующие факторы;
- вручную повторять типовые сценарии: одинаковые блюда, шаблоны, список покупок, недельный план.

Система должна сократить количество ручных действий и дать пользователю не просто журнал, а рабочую среду для контроля питания.

## Границы системы

В систему входят:

- учетные записи пользователей;
- роли `user` и `admin`;
- каталог продуктов;
- приемы пищи, шаблоны и составные рецепты;
- дневные цели по КБЖУ;
- аналитика за день и по периоду;
- вода, самочувствие, замеры тела;
- план питания и список покупок;
- экспорт отчетов и API-документация.

За границами системы остаются:

- медицинская диагностика;
- расчет лечебных диет по нормативам клиник;
- интеграция с внешними пищевыми базами;
- мобильные push-уведомления и нативные приложения;
- онлайн-оплата и коммерческие подписки.

## Участники предметной области

### Пользователь

Основной актор. Формирует личный пищевой дневник, задает цели, анализирует результаты и использует дополнительные инструменты планирования.

### Администратор

Управляет справочником продуктов и проверяет корректность справочных данных. Не должен получать неограниченный доступ к приватным пользовательским данным вне предусмотренных системой ролей и маршрутов.

### Проверяющий или преподаватель

Рассматривается как внешний наблюдатель, который оценивает полноту сценариев, корректность архитектуры, тестирования и структуру проекта.

## Основные пользовательские сценарии

1. Пользователь регистрируется или входит в систему.
2. Пользователь задает целевые значения по КБЖУ или применяет готовый пресет целей.
3. Пользователь добавляет прием пищи вручную или использует шаблон, рецепт или избранный продукт.
4. Система пересчитывает дневную сводку, прогресс по целям и рекомендации.
5. Пользователь фиксирует воду, самочувствие и замеры тела.
6. Пользователь формирует недельный план, рецепты и список покупок.
7. Администратор управляет справочником продуктов.
8. Пользователь экспортирует отчет и использует Swagger для проверки API.

## Ключевые сущности

### Пользователь и безопасность

- `User` — учетная запись, содержащая имя, email, пароль в хешированном виде и роль.
- `Goal` — персональные дневные цели пользователя по калориям и макронутриентам.

### Питание

- `Product` — справочная сущность с пищевой ценностью на 100 грамм.
- `Meal` — факт приема пищи в конкретную дату и время.
- `MealTemplate` — шаблон типового приема пищи.
- `Recipe` — составное блюдо, собранное из нескольких продуктов.
- `RecipeItem` — ингредиент рецепта.
- `MealPlan` — запланированный прием пищи на дату и время.

### Поведенческие и вспомогательные данные

- `HydrationLog` — запись воды.
- `Checkin` — отметка самочувствия.
- `BodyMetric` — замер тела.
- `ShoppingItem` — позиция списка покупок.
- `DayNote` — заметка дня.
- `FavoriteProduct` и `FavoriteTemplate` — избранные элементы для быстрого доступа.

## Бизнес-правила

1. Все числовые показатели КБЖУ, веса, воды и замеров должны быть неотрицательными.
2. Пользователь должен работать только со своими сущностями: приемами пищи, планами, замерами, заметками и шаблонами.
3. Администраторские маршруты должны быть отделены ролевой проверкой.
4. Дневная аналитика строится автоматически на основе фактических записей за выбранную дату.
5. Рецепт должен содержать хотя бы один ингредиент.
6. Генерация недельного плана выполняется на основе существующих шаблонов и рецептов.
7. Экспорт отчета должен отражать актуальные данные выбранной даты.

## Нефункциональные требования

- понятная модульная структура проекта;
- воспроизводимый локальный запуск;
- контейнеризация через Docker;
- документация API через Swagger;
- наличие тестов и негативных сценариев;
- возможность последующего облачного развертывания;
- соответствие принципам методологии Twelve-Factor App.

## Риски предметной области

- пользователь может вносить недостоверные или неполные данные;
- ошибки в справочнике продуктов искажают аналитику;
- отсутствие четкой ролевой модели приводит к утечке данных;
- чрезмерная сложность интерфейса снижает полезность приложения;
- отсутствие seed-данных и тестов затрудняет демонстрацию проекта.

## Вывод

Предметная область хорошо подходит для клиент-серверного CRUD-приложения: здесь есть несколько независимых сущностей, связанные пользовательские сценарии, роли, аналитика, планирование и понятные точки интеграции между frontend, backend и базой данных. Проект закрывает прикладную задачу контроля питания и одновременно дает достаточную инженерную глубину для курсового проектирования.
