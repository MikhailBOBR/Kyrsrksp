# Архитектура приложения

## Базовый подход

Для проекта выбрана клиент-серверная архитектура. Клиент отвечает за пользовательский интерфейс и взаимодействие с API, сервер отвечает за аутентификацию, бизнес-логику, валидацию данных и работу с базой данных.

## Предварительная схема компонентов

- клиентское приложение;
- HTTP API;
- сервис аутентификации;
- сервис расчёта КБЖУ;
- слой доступа к данным;
- реляционная база данных.

## Предварительные технические решения

- формат взаимодействия: REST API;
- формат данных: JSON;
- аутентификация: токены или сессии, окончательно определить после выбора стека;
- хранение данных: реляционная СУБД;
- контейнеризация: Docker.

## Нефункциональные требования

- корректная ролевая модель;
- базовая защита приватных данных пользователя;
- предсказуемая структура проекта;
- возможность локального запуска в контейнерах;
- удобство последующего развёртывания в облаке.
